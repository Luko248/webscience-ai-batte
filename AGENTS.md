# Repository Guidelines

## Project Structure & Module Organization
This is a small Astro site deployed to GitHub Pages.

- `src/pages/`: Route entrypoints. Keep page folders in kebab-case (for example, `festival-task/`, `encyclopedy-task/`).
- `src/layouts/`: Shared Astro layouts (for example, `Layout.astro`).
- `src/styles/`: Global styling (`global.css`) and theme variables.
- `public/`: Static assets served as-is (for example, `favicon.svg`).
- `.github/workflows/static.yml`: CI/CD pipeline for build and Pages deployment.
- `dist/`: Build output generated by `astro build` (do not hand-edit).

## Build, Test, and Development Commands
Use Bun for local workflows:

- `bun install`: Install dependencies.
- `bun run dev`: Start local dev server with hot reload.
- `bun run build`: Produce production build into `dist/`.
- `bun run preview`: Serve the built output locally for verification.

CI on `main` also runs `bun install` + `bun run build` before deploy.

## Coding Style & Naming Conventions
- Use TypeScript-friendly Astro patterns (`<script>` frontmatter with typed props where relevant).
- Follow existing formatting: 2-space indentation, semicolons in TS/JS, single quotes in scripts.
- Use `PascalCase` for reusable layout/component files (for example, `Layout.astro`).
- Use descriptive kebab-case for route folders and URL slugs.
- Keep Tailwind utility usage readable; extract repeated structure into layouts/components when it starts repeating.

## Testing Guidelines
There is currently no dedicated automated test framework configured.

- Minimum check before PR: run `bun run build` and ensure it succeeds.
- For UI changes, run `bun run dev` and validate affected routes manually.
- If you add a test framework, place tests near features (for example, `src/**/__tests__/`) and document new commands in `package.json`.

## Commit & Pull Request Guidelines
Current history uses short, imperative commit messages (for example, `Fix typo in base URL path configuration`).

- Keep commits focused and scoped to one logical change.
- Use imperative subject lines; mention affected area when helpful.
- PRs should include: purpose, key changes, manual verification steps, and screenshots for visual updates.
- Link related issues/tasks and note any deployment-impacting config changes (especially `astro.config.mjs` base path or Pages workflow).
